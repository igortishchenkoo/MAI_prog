Текстовый файтинг на C++ — конспект к защите

Проект: пошаговый консольный файтинг «Воин против Мага». Один файл main.cpp, C++17, Visual Studio. Здесь — всё, что есть в коде: каждое понятие с определением, для сложного — пояснение «просто» и «умно», где оно в коде, и вопросы-ловушки в конце.

1Что вообще в программе

Игра: два игрока (хотсит) выбирают класс (Воин / Маг) и броню, потом по очереди бьют друг друга атаками, пока у кого-то не кончится здоровье. Урон случайный, зависит от матрицы «атака × броня цели», часть урона срезает резист брони, можно увернуться. У воина ресурс — ярость (копится в бою, тратится на ульту), у мага — мана (тратится на заклинания, восстанавливается каждый ход).

Карта программы: main → фабрика и бой; иерархия классов; пассивные данные и утилиты.

2ООП: главные понятия

Класс и объект

Определение. Класс — пользовательский тип данных, объединяющий данные (поля) и функции для работы с ними (методы). Объект — конкретный экземпляр класса в памяти.

Просто: класс — чертёж дома, объект — построенный дом. По одному чертежу можно построить сколько угодно домов, у каждого свой адрес и своя мебель.
Где в коде: класс Character; объекты — то, что создаётся в createFighter() через new Warrior(...).

Инкапсуляция (public / protected / private)

Определение. Сокрытие внутреннего состояния объекта: данные закрыты, доступ к ним — только через публичные методы. Модификаторы: public — видно всем; protected — видно классу и его наследникам; private — видно только самому классу.

Просто: банкомат. Деньги внутри (private), но снаружи есть кнопки (public-методы). Ты не можешь залезть рукой в сейф и «поправить» сумму — только через операции, которые проверяют правила.
Умно: инкапсуляция защищает инварианты класса — условия, которые всегда должны выполняться. Например, «HP не бывает меньше 0» гарантируется тем, что менять health можно только через takeDamage(), где стоит проверка. Если бы поле было public, любой код мог бы записать туда −50 и сломать логику.
Где в коде: поля health, armor, attacks лежат в protected (не private — чтобы наследники Warrior/Mage могли их видеть), а снаружи доступ только через методы.

Конструктор и список инициализации

Определение. Конструктор — специальный метод, вызываемый при создании объекта; имя совпадает с именем класса, возвращаемого типа нет. Список инициализации — конструкция : поле(значение), ... после заголовка конструктора, инициализирующая поля до входа в тело.

Character(std::string name, int health, Armor armor)
    : name(name), health(health), maxHealth(health), armor(armor) {}
Просто: это анкета при рождении объекта: имя, здоровье, броня записываются сразу, а не «родился пустым, потом заполним».
Умно: поля инициализируются в порядке их объявления в классе, а не в порядке записи в списке. Поэтому список надо писать в том же порядке, что и объявления, иначе при /W4 будет предупреждение, а в хитрых случаях (одно поле зависит от другого) — баг. Заметь трюк maxHealth(health): максимум задаётся от стартового здоровья, второй параметр не нужен.
Тонкость: конструктор наследника обязан вызвать конструктор базы: Warrior(...) : Character(name, health, armor) {...} — данные «передаются вверх». Без этого база не соберётся.

Наследование

Определение. Механизм создания нового класса на основе существующего: наследник получает все поля и методы базового класса и может добавлять свои или переопределять виртуальные. Запись: class Warrior : public Character. Публичное наследование выражает отношение «is-a»: воин есть персонаж.

Просто: «воин — это персонаж со своими особенностями». Всё общее (HP, броня, умение получать урон) написано один раз в Character, а Warrior и Mage добавляют только своё: ярость и ману.
Умно: наследование — инструмент устранения дублирования и построения полиморфных иерархий. Альтернатива — композиция («has-a»); правило хорошего тона: наследование только там, где реально «is-a» и нужен полиморфизм, иначе — композиция. У нас ровно этот случай: бой работает с «каким-то Character», не зная каким.
Где в коде: class Warrior : public Character, class Mage : public Character.

Полиморфизм, virtual и override

Определение. Полиморфизм — возможность работать с объектами разных классов через единый интерфейс базового класса, при этом вызывается версия метода реального класса объекта. Включается словом virtual в базе; наследник помечает свою версию override.

virtual std::string getClassName() const { return "Боец"; }   // база
std::string getClassName() const override { return "Воин"; }  // Warrior
Просто: кнопка «подай голос» у животных: жмёшь её у собаки — «гав», у кота — «мяу». Код, который жмёт кнопку, один и тот же, а результат зависит от того, кто перед тобой.
Умно: это динамический полиморфизм: решение, какую функцию звать, принимается во время выполнения через таблицу виртуальных функций (vtable). У каждого полиморфного объекта есть скрытый указатель (vptr) на vtable его класса; вызов p->getClassName() идёт через эту таблицу. Поэтому Character* p = new Warrior(...) честно отвечает «Воин». override — страховка компилятора: если опечатался в сигнатуре, получишь ошибку, а не тихо созданный новый метод.
Где в коде: getClassName, showResource, canAfford, payFor, onDealDamage, onTakeDamage, onTurnStart — все виртуальные; runBattle и attackTarget работают с Character&, не зная, воин там или маг.

Виртуальный деструктор

Определение. Деструктор — метод, вызываемый при уничтожении объекта (~Character()). Если класс используется полиморфно (удаляется через указатель на базу), деструктор обязан быть виртуальным.

virtual ~Character() = default;
Просто: ты сдаёшь номер в отеле по карточке «постоялец» (указатель на базу). Если деструктор не виртуальный, уборка пройдёт только по общему списку, а личные вещи конкретного гостя (данные наследника) останутся гнить.
Умно: delete p1, где p1 имеет тип Character*, а объект на самом деле Warrior — при невиртуальном деструкторе это неопределённое поведение (UB): вызовется только деструктор базы. С virtual вызов уходит через vtable в правильный деструктор наследника, потом автоматически в базовый. Правило: есть хоть один virtual-метод — делай и деструктор виртуальным. = default означает «сгенерируй стандартный».
Где в коде: в Character; критично, потому что в main есть delete p1; delete p2; по указателям Character*.

struct против class

Определение. В C++ struct и class — одно и то же, кроме доступа по умолчанию: в struct всё public, в class всё private. По соглашению struct используют для простых «пакетов данных», class — когда есть инкапсуляция и поведение.

Где в коде: struct Attack — чистые данные атаки (имя, урон, множители, стоимость) с одним вспомогательным методом multiplierVS; class Character — полноценный класс с закрытым состоянием.

enum class (перечисление)

Определение. enum class — типобезопасное перечисление: свой тип с фиксированным набором именованных значений, требует префикс (Armor::Heavy) и не преобразуется неявно в int.

enum class Armor { Heavy, Medium, Light };
Просто: вместо загадочного «броня = 1» (что это? тяжёлая? средняя?) код говорит словами: Armor::Heavy. Опечататься или передать «броню = 47» невозможно — компилятор не пустит.
Умно: старый enum (без class) сливает имена в общую область видимости и неявно кастуется в int — источник конфликтов и багов. enum class «огорожен»: имена внутри, преобразование только явное. Поэтому для печати нужна функция-переводчик armorToString со switch.
Где в коде: Armor; по нему работают armorToString, dodgeChance, damageResist, multiplierVS.

const-корректность

Определение. Метод с const после скобок (bool isAlive() const) обещает не менять объект; компилятор это проверяет. Параметр const T& — «передай без копии, но менять не буду».

Просто: табличка «руками не трогать». isAlive() только смотрит на здоровье — пусть компилятор гарантирует, что оно случайно не изменится.
Умно: const — часть контракта интерфейса: const-методы можно звать у const-объектов и по const-ссылкам (как в drawBoard(const Character& p1, ...)). Убери const у print() — и drawBoard перестанет компилироваться. Ещё нюанс: payFor в базе объявлен без const (он меняет ресурс) — значит и переопределения в наследниках должны быть без const, иначе это не override, а другой метод.
Где в коде: isAlive, print, getClassName, showResource, canAfford, listAttacks, showAttacks, askAttackIndex, multiplierVS — const; takeDamage, payFor, useAttack — нет, они меняют состояние.

Хуки (виртуальные точки расширения)

Определение. Приём: базовый класс в нужных местах алгоритма зовёт виртуальные методы с пустой реализацией по умолчанию; наследники переопределяют только нужные. Родственник паттерна «Шаблонный метод».

virtual void onDealDamage(int dmg) {}   // база: ничего
void onDealDamage(int dmg) override { addRage(5); }  // Warrior: +ярость
Просто: в сценарии боя оставлены «розетки»: momento удара, момент получения урона, начало хода. Воин втыкает в них ярость, маг — реген маны. Кому не надо — оставляет пустыми.
Умно: так общий алгоритм (attackTarget, takeDamage, цикл хода) написан один раз и не знает про классы, а специфика классов «подписывается на события». Это удобнее, чем if-ы вида «если воин — добавь ярость»: добавление третьего класса не требует менять движок. Пары canAfford/payFor — те же хуки для экономики ресурсов: у мага мана, у воина ярость, база по умолчанию «всё бесплатно».
Где в коде: onDealDamage, onTakeDamage, onTurnStart, canAfford, payFor; дергаются из attackTarget, takeDamage, runBattle, useAttack, askAttackIndex.

3Память: указатели, ссылки, new/delete

Указатели, new и delete

Определение. Указатель (Character*) — переменная, хранящая адрес объекта. new создаёт объект в динамической памяти (куче) и возвращает адрес; delete уничтожает объект и освобождает память. *p — разыменование, доступ к самому объекту.

Просто: указатель — бумажка с адресом дома, а не сам дом. new строит дом и даёт адрес, delete — сносит по адресу. Потерял бумажку, не снеся дом, — дом остался стоять и занимать место (утечка памяти).
Умно: указатель на базу нужен нам ради полиморфизма: тип создаваемого бойца известен только в рантайме (игрок выбирает в меню), а Character* p может держать любого наследника. Каждому new — ровно один delete, иначе утечка. В современном C++ это автоматизируют умные указатели (std::unique_ptr сам зовёт delete) — честный ответ на вопрос «как сделали бы в продакшене».
Где в коде: createFighter() возвращает Character* из new Warrior/new Mage; в main — runBattle(*p1, *p2) и парные delete.

Ссылки (&) и передача параметров

Определение. Ссылка (Character&) — псевдоним существующего объекта. Передача по значению копирует объект; по ссылке — работаем с оригиналом; по const-ссылке — без копии и без права менять.

Просто: передать по значению — дать ксерокопию тетради: что в ней ни черкай, оригинал чист. По ссылке — дать саму тетрадь. Когда воин бьёт мага, урон должен получить настоящий маг, а не его ксерокопия — поэтому цель передаётся как Character& target.
Умно: ссылка обязана быть привязана при создании и не бывает «нулевой» — этим она безопаснее указателя. Правило выбора: менять объект → T&; только читать, объект тяжёлый → const T& (без копирования); мелкие типы (int, enum) — по значению. Передача по значению полиморфного объекта опасна ещё и срезкой (см. ловушки).
Где в коде: attackTarget(Character& target, ...), runBattle(Character& p1, Character& p2), drawBoard(const Character&, const Character&), range-for const Attack& atk.

4Стандартная библиотека

std::vector и range-for

Определение. std::vector<T> — динамический массив: хранит элементы одного типа, сам управляет памятью и размером. push_back добавляет в конец, size() — количество, v[i] — доступ по индексу (без проверки границ!). Range-for (for (const Attack& a : attacks)) — обход всех элементов по очереди.

Просто: резиновый список: кинул четвёртую атаку — сам растянулся. А range-for читается как «для каждой атаки из списка».
Умно: v[i] с плохим индексом — неопределённое поведение (выход за границы), поэтому весь пользовательский ввод индексов у нас зажат через readInt(1, size) − 1. size() возвращает беззнаковый size_t — при сравнении с int нужен static_cast<int>, иначе предупреждение о сравнении знакового с беззнаковым. В range-for берём элемент по const&, чтобы не копировать структуру на каждой итерации.
Где в коде: std::vector<Attack> attacks; заполняется списком в фигурных скобках в конструкторах Warrior/Mage; обходится в showAttacks/listAttacks.

Случайные числа: <random>

Определение. Современная случайность в C++ = движок + распределение. std::mt19937 — генератор (Mersenne Twister), засевается один раз из std::random_device; std::uniform_int_distribution<int>(a, b) — равномерно раскладывает вывод движка в диапазон [a; b] включительно.

std::mt19937 rng(std::random_device{}());
int rollDamage(int minDmg, int maxDmg) {
    std::uniform_int_distribution<int> dist(minDmg, maxDmg);
    return dist(rng);
}
Просто: движок — барабан лототрона, который крутится и выдаёт сырые числа; распределение — трафарет, который превращает их в честные «от 5 до 15». А зерно из random_device — то, что каждый запуск игры делает разным.
Умно: почему не rand() % 11 + 5: у rand слабый генератор и, главное, взятие остатка даёт перекос (modulo bias) — младшие значения выпадают чаще, если диапазон rand не делится нацело. Движок глобальный и один, потому что он хранит состояние: пересоздавать его на каждый бросок — значит пересевать и убивать статистику. Запись std::random_device{}() — создали временный объект и сразу вызвали его operator(), получив одно зерно. Шанс события: tryRoll(p) — бросок 1..100 и сравнение <= p.

Строки, потоки и защита ввода

Определение. std::getline(std::cin, line) читает всю строку в std::string. std::stringstream — поток над строкой в памяти: из него можно читать так же, как из cin. На этом построен защищённый ввод:

int readInt(int minVal, int maxVal) {
    while (true) {
        std::string line;
        if (!std::getline(std::cin, line)) std::exit(0);
        std::stringstream ss(line);
        int value; std::string extra;
        if (ss >> value && !(ss >> extra) && value >= minVal && value <= maxVal)
            return value;
        std::cout << "Введи число от " << minVal << " до " << maxVal << ": ";
    }
}
Просто: раньше буква в поле числа ломала игру: программа входила в бесконечный цикл ошибок. Теперь мы сначала забираем ввод целиком как текст (текст сломать нельзя), а потом аккуратно проверяем: это число? после него нет мусора? оно в диапазоне? Нет — вежливо переспрашиваем.
Умно: прямое cin >> x при нечисловом вводе ставит поток в fail state (все следующие чтения мгновенно проваливаются) и оставляет мусор в буфере — отсюда классический вечный цикл. Классическое лечение cin.clear() + cin.ignore() надо повторять у каждого чтения; наш способ убирает проблему архитектурно: парс-ошибки портят только временный stringstream. Три проверки: ss >> value — число извлеклось; !(ss >> extra) — после числа пусто (отсекает «5абв»); диапазон. Проверка !std::getline(...) ловит EOF (Ctrl+Z) — выходим, а не крутимся вечно.

5Игровая логика и приёмы

Атаки как данные + матрица урона

Идея. Атака — не функция с if-ами, а запись struct Attack: имя, разброс урона, три множителя (против тяжёлой/средней/лёгкой), стоимость. Итоговый урон = бросок × множитель по броне цели × (1 − резист брони цели).

Просто: «камень-ножницы-бумага»: физика воина буксует о латы, но рвёт лёгкую броню; магия наоборот плавит латников. Все правила лежат в табличке, а не размазаны по коду.
Умно: это «data-driven» подход: добавить новую атаку = добавить одну строку данных, движок менять не надо. Матрица и резист — два разных слоя: матрица отвечает за matchup конкретной атаки, резист — за общую «толщину» брони. Баланс (95 HP воина, 85 HP + реген 7 у мага, резист 20/10/0) подобран симуляцией ~100 000 автоматических боёв до ~50/50.

Уворот и шансы

Идея. Шанс уворота зависит от брони владельца: тяжёлая 5%, средняя 15%, лёгкая 30%. Перед уроном бросается tryRoll(dodgeChance(target.armor)): успех — атака промахнулась, выходим из attackTarget через ранний return.

Тонкость: ресурс списывается в useAttack до проверки уворота. То есть промахнувшаяся ульта всё равно сжигает 50 ярости. Это осознанное упрощение — и готовый ответ на вопрос «что бы ты улучшил» (варианты: списывать после попадания, или дать ульте свойство «не мажет»).

Ресурсы: ярость и мана

Идея. Экономики зеркальные: мага мана «сверху вниз» (старт 100, заклинания тратят, +7 в начале хода через onTurnStart, потолок 100), воина ярость «снизу вверх» (старт 0, +5 за нанесённый удар, +10 за полученный — через хуки, потолок 50, ульта стоит 50).

Просто: маг начинает бой богачом и постепенно беднеет, воин — злится от драки и к середине боя взрывается ультой. Потолок ярости = цене ульты: накопил — разрядился — копи заново.
Где в коде: canAfford/payFor (можно ли платить и как), onDealDamage/onTakeDamage (накопление ярости), onTurnStart (реген маны), приватный addRage с капом.

Фабрика бойцов и меню

Идея. createFighter() — фабричная функция: спрашивает класс и броню и возвращает готовый объект как Character*. Ограничение из задания зашито в саму структуру меню: воину предлагаются только тяжёлая/средняя, магу — средняя/лёгкая: запрещённый вариант просто не существует на экране.

Умно: «фабрика» — паттерн создания: вызывающий код (main) не знает, какой конкретно класс родился, и работает с ним через базовый интерфейс. Внутри — тернарный оператор условие ? да : нет для выбора брони. Нумерация меню — человеческая, с 1: игрок жмёт 1..N, код делает readInt(1, N) − 1 и работает с индексами 0..N−1.

Разделение логики и вывода, целочисленная математика

Идея. Весь «экран» собирается в drawBoard (очистка + табло двух бойцов) и healthBar; боевые правила вывода не касаются. Полоска здоровья: filled = cur * total / max.

Тонкость: порядок операций важен: cur / max * total в целых числах даст 0 почти всегда (51/85 = 0), поэтому сначала умножаем, потом делим. И static_cast<int> при уроне отбрасывает дробь (5.9 → 5), а не округляет — явный каст ещё и глушит предупреждение о потере точности.

6Вопросы-ловушки (какие могут задать на защите)

Почему деструктор Character виртуальный? Что случится, если убрать virtual?
delete p1 идёт через Character*, а объект — Warrior. Без virtual вызовется только деструктор базы — это неопределённое поведение, ресурсы наследника не освободятся корректно. С virtual вызов уходит в правильный деструктор через vtable.
Чем отличается перегрузка (overload) от переопределения (override)?
Перегрузка — несколько функций с одним именем, но разными параметрами, выбирается на этапе компиляции. Переопределение — наследник заменяет виртуальный метод базы с той же сигнатурой, выбирается в рантайме через vtable. У нас переопределение: getClassName в Warrior/Mage.
Почему поля в protected, а не private? А почему не public?
private скрыл бы health/armor/attacks даже от Warrior/Mage, которым они нужны. public сломал бы инкапсуляцию — кто угодно мог бы записать в health мусор мимо takeDamage. protected — середина: наследникам можно, чужим нельзя.
В attackTarget вы читаете target.armor — но armor же protected, а target чужой объект?
Доступ в C++ проверяется на уровне класса, а не объекта: метод класса Character имеет доступ к protected/private-полям любого объекта Character (и наследников через тип базы). Поэтому target.armor внутри метода Character легален.
Что такое срезка (slicing)? Где она могла бы случиться?
Если полиморфный объект передать/присвоить по значению как базу — Character c = *p1; — скопируется только «базовая часть», ярость воина и его vtable-поведение потеряются: c.getClassName() скажет «Боец». Поэтому везде передаём по ссылке (Character&) или указателю.
Почему нельзя просто rand() % 11 + 5?
Два греха: слабый генератор и modulo bias — если диапазон rand() не делится нацело на 11, младшие значения выпадают чаще, разброс нечестный. uniform_int_distribution математически честный, mt19937 — качественный движок.
Почему движок rng глобальный и один, а распределение создаётся в функции каждый раз?
Движок хранит состояние и должен «крутиться» непрерывно; пересоздание = пересев = плохая случайность (и дорого). Распределение — лёгкий объект без важного состояния, создавать его дёшево.
Что было бы, если игрок вводит «абв» при старом вводе cin >> x?
Поток вставал в fail state, «абв» оставался в буфере, каждый следующий >> мгновенно проваливался — бесконечный цикл ошибок. Решение: читать всю строку getline (не фейлится), парсить через stringstream, при ошибке переспрашивать. Плюс проверка !(ss >> extra) отсекает «5абв», а проверка результата getline — EOF.
Почему в списке инициализации maxHealth(health) работает, хотя maxHealth объявлен после health?
Поля инициализируются в порядке объявления: сначала health (из параметра), потом maxHealth — от уже готового поля... на самом деле от параметра health (имена совпадают, берётся параметр). Важно правило: порядок инициализации определяется порядком объявления полей, поэтому список пишем в том же порядке — иначе можно случайно читать ещё не созданное поле.
Чем enum class лучше обычного enum?
Не сорит именами в общую область видимости (нужен префикс Armor::), не преобразуется неявно в int, нельзя сравнить с другим enum — типобезопасность. Минус-следствие: для печати нужна своя функция-переводчик (armorToString).
Зачем static_cast, нельзя ли просто присвоить double в int?
Неявное сужение работает, но компилятор при /W4 предупреждает о потере точности. static_cast — явное «я понимаю, что отбрасываю дробь». Важно: отбрасывает, а не округляет: 13.9 → 13.
Промах по увороту — а ресурс атаки уже потрачен. Это баг?
Осознанное упрощение: payFor вызывается в useAttack до attackTarget. Знаю об этом; варианты починки — платить после попадания либо дать ульте флаг «без промаха». Хороший кандидат на доработку.
Почему createFighter возвращает указатель, а не объект Character?
Тип бойца известен только в рантайме (выбор игрока), а вернуть «по значению как Character» = срезка. Указатель на базу хранит любого наследника без потерь и включает полиморфизм. Цена — обязанность вызвать delete (в проде — unique_ptr).
Что делает SetConsoleOutputCP(1251)?
Говорит консоли Windows интерпретировать байты вывода как кодировку Windows-1251 — в ней сохранён исходник, иначе кириллица превращается в кракозябры. Парная SetConsoleCP — про ввод. Альтернатива — всё в UTF-8 (chcp 65001 + флаг /utf-8).
Почему в меню игрок жмёт с 1, а вектору нужен индекс с 0?
Людям привычна нумерация с 1, массивам — с 0. Правило: интерфейс человеческий, индексация машинная; на границе — конверсия readInt(1, N) − 1, и диапазон проверен до обращения к вектору (защита от выхода за границы).
